# Milestone 4 Plan (Block System)

## 목표
`roadmap.md`의 **M4. Block System** 항목을 현재 캔버스 구조 위에 맞게 구현 가능한 형태로 정리한다.
- `Add Node`로 메뉴 노드를 생성하고, 생성된 노드 위에서 블록 타입을 확정하는 플로우 구현
- 텍스트/이미지/링크 블록 편집 UX 구현
- 음악/게임/영화/책 블록 내부 검색 선택 UX 구현
- 호버/편집 중/도킹 가능 여부/스냅 프리뷰 시각 상태 정리

## 범위 (In Scope)
- `Add Node` 클릭 시 일반 블록이 아니라 타입 선택용 메뉴 노드 생성
- 메뉴 노드에서 블록 타입을 선택하면 해당 타입의 실제 블록 노드로 전환
- 편집 UI는 노드 내부에서 처리
- 텍스트/이미지/링크 블록의 제한된 편집 필드 반영
- 음악/게임/영화/책 블록의 내부 검색/결과 선택 UX
- 노드 카드의 시각 상태 체계화(`default`, `hover`, `editing`, `preview`, `invalid`)
- M4 범위 테스트(Unit + Component 중심)

## 범위 (Out of Scope)
- 실제 외부 API 호출, 인증, 쿼터, 에러 정책(M5)
- 저장 어댑터, 협업, 권한 정책(M6)
- 전역 선택 패널, inspector 사이드바, 멀티 셀렉트
- 리치 텍스트 에디터, 이미지 업로드 저장소, 링크 unfurl 고도화
- 복잡한 registry/plugin 시스템

## 구현 단계

### 1. 블록 데이터와 메뉴 노드 경계 정리
- 저장 대상인 `BlockData`와 런타임 전용 메뉴 노드 상태를 분리한다.
- 저장 대상 블록 타입은 두 그룹으로 유지한다.
  - 직접 입력형: `text`, `image`, `link`
  - 검색 선택형: `music`, `game`, `movie`, `book`
- 메뉴 노드는 블록 타입 미정 상태를 표현하는 UI용 노드로만 사용하고 저장 모델에는 포함하지 않는다.
- 검색 선택형 블록은 M4에서 필요한 표준 필드만 유지한다.
  - 예: `title`, `description`, `thumbnailUrl`, `externalUrl`, `meta`
- `createDefaultBlockData`, `validateBlockData`, `getBlockDisplayText`는 실제 블록 타입만 기준으로 재정리한다.

완료 기준
- 메뉴 노드와 실제 블록 데이터가 타입 수준에서 구분된다.
- 저장/복원 모델에 임시 UI 상태가 섞이지 않는다.

### 2. 메뉴 노드 생성 플로우 도입
- 상단 `Add Node` 버튼은 실제 블록을 바로 만들지 않고 메뉴 노드를 생성한다.
- 메뉴 노드는 현재 stage 기준 기본 위치에 생성한다.
- 생성 직후 메뉴 노드는 즉시 타입 선택 가능한 상태여야 한다.
- 메뉴 노드 위에서 블록 타입을 선택하면 다음 순서로 전환한다.
  1. 타입 결정
  2. 타입별 기본 데이터 생성
  3. 동일 위치에서 실제 블록 노드로 치환
  4. 필요한 경우 해당 노드 편집 UI 즉시 오픈

완료 기준
- `Add Node` 한 번으로 "빈 메뉴 노드 생성 -> 타입 확정" 흐름이 끊기지 않는다.
- 블록 타입 선택 전까지는 임시 노드라는 사실이 UI로 명확히 드러난다.

### 3. 전역 선택 없이 노드 단위 편집 흐름 정리
- "선택된 노드" 상태는 도입하지 않는다.
- 대신 노드 단위의 `editing/open` 상태만 최소 범위로 둔다.
- 편집 진입 규칙을 고정한다.
  - 메뉴 노드 생성 직후: 타입 선택 UI 오픈
  - 실제 블록 노드 클릭 또는 명시적 edit affordance: 해당 노드 편집 UI 오픈
  - 빈 캔버스 클릭, 드래그 시작, 다른 노드 편집 진입: 현재 편집 UI 닫기
- 편집 UI는 `SquareNode` 내부 또는 노드 인접 popover/sheet로 처리하되, 캔버스 바깥 inspector는 두지 않는다.

완료 기준
- 전역 선택 모델 없이도 어느 노드를 편집 중인지 드러난다.
- 드래그/도킹과 편집 UI가 충돌하지 않는다.

### 4. 직접 입력형 블록 편집 UX
- 텍스트 블록은 `text`만 편집 가능하게 한다.
- 이미지 블록은 `imageUrl`, `description`만 편집 가능하게 한다.
- 링크 블록은 `url`만 편집 가능하게 한다.
- 편집값 반영은 "입력 -> node.data 업데이트 -> 카드 요약 갱신" 흐름으로 유지한다.
- `title`은 사용자 입력 필드로 노출하지 않고 표시용 기본값 또는 파생값으로 관리한다.
  - text: `text` 기반 제목/요약
  - image: `description` 또는 기본 라벨 기반 제목/요약
  - link: `url` 기반 제목/요약
- 필수 필드가 비어 있으면 카드와 편집 UI에서 즉시 구분 가능해야 한다.

완료 기준
- 블록별 편집 가능 필드가 주석 요구사항과 정확히 일치한다.
- 입력 중에도 노드 카드가 최소 요약 정보를 안정적으로 보여준다.

### 5. 검색형 블록 내부 검색 선택 UX
- 음악/게임/영화/책 블록은 공통 검색 UI 흐름을 공유한다.
- 검색 UI는 해당 노드 내부 또는 노드 인접 오버레이에서 열려야 한다.
- M4에서는 실제 API 대신 fixture/stub 데이터 기반 검색 흐름만 제공한다.
  - 검색어 입력
  - 로딩 상태 또는 검색 중 표시
  - 결과 목록
  - 결과 선택
  - 선택 결과를 블록 표준 필드로 반영
- 선택 완료 후에는 검색 결과가 카드 요약에 반영되고, 같은 노드에서 다시 검색을 열어 재선택할 수 있어야 한다.

완료 기준
- 검색형 4개 블록이 공통 인터랙션으로 생성/선택된다.
- M5에서는 데이터 공급부만 교체하면 되도록 경계가 분리된다.

### 6. 노드 카드 레이아웃과 시각 상태 정리
- `SquareNode`를 단순 텍스트 박스에서 블록 카드 + 메뉴 노드 UI로 확장한다.
- 카드 내부 정보 우선순위를 정한다.
  - 메뉴 노드: 타입 선택 CTA
  - text/link: 대표 텍스트 1순위
  - image/search형: 썸네일 또는 제목 1순위
- 시각 상태 토큰을 분리한다.
  - 기본
  - hover
  - editing/open
  - snap preview target
  - invalid dock / fallback 직후 경고
- 편집 중 강조와 도킹 프리뷰가 충돌하지 않도록 우선순위를 고정한다.

완료 기준
- 메뉴 노드와 실제 블록 노드가 시각적으로 즉시 구분된다.
- 사용자가 편집 상태와 배치 상태를 혼동하지 않는다.

### 7. 상태 경계와 컴포넌트 구조 정리
- `CanvasCoreInner`에 몰린 책임을 다음 수준으로 나눈다.
  - 캔버스 상호작용: 노드/엣지/도킹/메뉴 노드 생성
  - 노드 UI 상태: 열린 편집기, 열린 검색 UI, 메뉴 노드 전환
  - 블록 UI: 타입 선택기, 직접 입력형 편집기, 검색 결과 리스트, 카드 렌더
- 블록 편집 데이터는 가능하면 `node.data`를 source of truth로 유지한다.
- 메뉴 노드와 편집 UI open/close 상태만 별도 runtime UI state로 둔다.
- 검색 데이터 공급 방식은 단순 fixture/stub 함수로 시작하고, M5에서 API client adapter로 치환 가능하게 둔다.

완료 기준
- `CanvasCoreInner`가 오케스트레이션 중심으로 단순화된다.
- 저장 대상 데이터와 런타임 UI 상태가 명확히 분리된다.

### 8. 검증 및 회귀 점검
- Unit(Vitest)
  - 블록 기본값 생성
  - 블록 validation/fallback
  - 메뉴 노드 -> 실제 블록 전환 규칙
  - 검색 결과 -> 표준 블록 데이터 매핑
- Component(RTL)
  - `Add Node`로 메뉴 노드 생성
  - 메뉴 노드에서 타입 선택 후 실제 블록 전환
  - text/image/link 편집 필드 제한 검증
  - 검색형 블록 결과 선택 반영
  - hover/editing/preview 상태 클래스 검증
- 수동 회귀
  - M1/M2의 드래그, 도킹, stage 확장, edge 연결 회귀 확인

완료 기준
- 메뉴 노드 생성부터 블록 확정/편집까지 핵심 플로우가 자동 테스트로 커버된다.
- 기존 캔버스 상호작용 회귀가 없다.

## 산출물
- 메뉴 노드와 실제 블록 노드를 구분하는 런타임 모델
- 메뉴 노드 기반 블록 생성 UI
- 직접 입력형 블록 편집 UI
- 검색형 블록 내부 검색/선택 UI
- 상태별 블록 카드 시각 시스템

## 검증 체크리스트 (M4 기준)
- [ ] `Add Node`로 메뉴 노드가 생성되고, 생성된 노드 위에서 블록 타입을 선택할 수 있다.
- [ ] 텍스트/이미지/링크 블록이 요구된 필드만 편집 가능하다.
- [ ] 음악/게임/영화/책 블록 내부 검색 선택 UX가 동작한다.
- [ ] 호버/편집 중/도킹 가능 여부/스냅 프리뷰 시각 상태가 정리됐다.

## 권장 구현 순서
1. 메뉴 노드 runtime 모델 추가 + 블록 데이터 타입 확장
2. `Add Node -> 메뉴 노드 생성 -> 타입 선택 -> 실제 블록 전환` 흐름 구현
3. text/image/link 편집 UI 구현
4. 검색형 블록 fixture/stub 기반 검색 선택 UI 구현
5. 카드 시각 상태 정리 + 테스트/회귀 점검

## 리스크와 대응
- 리스크: 메뉴 노드와 실제 블록 노드가 같은 타입 체계에 섞이면 저장 모델이 오염될 수 있음
- 대응: 메뉴 노드는 런타임 전용 노드 타입으로 분리하고 직렬화 대상에서 제외

- 리스크: 전역 선택 없이 편집 UI를 열고 닫는 규칙이 불안정할 수 있음
- 대응: "동시에 하나의 노드만 편집 UI 오픈" 규칙과 닫힘 이벤트를 명시적으로 고정

- 리스크: 블록별 편집 필드 제한과 표시 필드가 어긋날 수 있음
- 대응: 편집 가능 필드와 카드 표시 규칙을 타입별로 함께 테스트해 회귀를 막음
