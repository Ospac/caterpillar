# Milestone 3 Plan (Data Model and Domain Rules)

## 목표
`roadmap.md`의 **M3. Data Model and Domain Rules**를 과하게 일반화하지 않고 최소 범위로 정리한다.
- 런타임 상태와 저장 모델의 경계만 명확히 분리
- M4에 필요한 최소 블록 데이터 구조만 정의
- 저장/복원 가능한 단순 document shape만 정리

## 범위 (In Scope)
- React Flow 런타임 상태와 저장 대상 필드 분리
- `node.data`를 블록 타입별로 확장 가능한 최소 구조로 정리
- 저장/복원에 필요한 문서 형태 정의
- 기본값 보정 수준의 간단 validation 규칙 정리

## 범위 (Out of Scope)
- 스키마 버전 관리
- 마이그레이션 체인 설계
- 과도하게 일반화된 registry hook 설계
- 실제 블록 편집 UI 구현(M4)
- 저장 어댑터 구현(M6)

## 구현 단계

### 1. 데이터 경계 정리
- 런타임 상태와 저장 상태를 구분한다.
- 저장해야 하는 값과 재계산 가능한 값을 나눈다.
- 타입 이름만 봐도 용도가 드러나게 정리한다.

완료 기준
- `CanvasCore`가 들고 있는 상태 중 저장 대상이 명확하다.
- 저장 모델이 React Flow 내부 구조에 과하게 끌려가지 않는다.

### 2. 최소 블록 데이터 구조 정의
- `label` 중심 구조를 M4 기준의 최소 블록 데이터로 확장한다.
- 공통 필드와 타입별 필드를 가볍게 나눈다.
- 새 블록 타입 추가 시 복잡한 추상화 없이 필드 확장이 가능해야 한다.

완료 기준
- 텍스트/이미지/링크 블록이 같은 규칙으로 표현된다.
- 블록 생성과 편집 UI가 기대할 필드를 바로 알 수 있다.

### 3. 단순 document shape 정리
- `nodes`, `edges`, `visibleStage` 중심의 저장 구조를 정한다.
- `viewport`나 임시 편집 상태는 제외한다.
- 저장과 복원은 현재 shape 하나만 기준으로 처리한다.

완료 기준
- 현재 문서 형태만으로 캔버스를 복원할 수 있다.
- 버전 필드나 migration 전제 없이도 구현이 단순하게 유지된다.

### 4. 기본 validation만 도입
- 블록 타입별 필수 필드를 정의한다.
- 누락 데이터는 기본값 보정 또는 로드 거부로 처리한다.
- 복잡한 validation 상태 모델은 도입하지 않는다.

완료 기준
- 저장 데이터 복원 실패 케이스가 예측 가능하다.
- M4 편집 UI와 같은 규칙을 재사용할 수 있다.

## 산출물
- 저장 모델 타입
- 최소 블록 데이터 구조
- 간단 validation 규칙
- 저장/복원 진입점
